1. Executive Summary

Critical Finding: Every new CVW asset (character, vehicle, or weapon) requires 1–3 weeks of Technical Art time for "UE Derivation" — the manual process of wiring UE content into BLAM tag structures. At M2/M3 scale (3–4× content volume), this represents 6–12 person-months of pure setup work per milestone, creating an unacceptable bottleneck.
Primary Recommendation: Invest in a Python Tag Field Read/Write API (R1). A working prototype already exists. This single capability unlocks automation of the entire UE Derivation setup process, reducing new asset creation from weeks to under one day. With parallel staffing (~3 eng + 3 TA), all short-and-medium-term recommendations ship within 5 months.

2. Current Architecture

2.1 Tag Data Asset System

Every BLAM tag type is represented in UE as a subclass of UBlamTagDataAssetBase — a UE DataAsset containing binary BLAM bulk data plus UE metadata. The system defines ~100+ tag types, with the following CVW-relevant types being central:

Tag ExtensionUE ClassRole
.modelUBlamModelTagDataAssetMaster tag for any dynamic object — owns all sub-tags
.collision_modelUBlamCollisionModelTagDataAssetPhysical collision (region/permutation/bone per mesh)
.skeleton_modelUBlamSkeletonModelTagDataAssetSkeleton/rig data
.physics_modelUBlamPhysicsModelTagDataAssetRagdoll/physics simulation
.render_modelUBlamRenderModelTagDataAssetVisual mesh
.model_animation_graphUBlamModelAnimationGraphTagDataAssetAnimation state machine
.bipedUBlamBipedTagDataAssetCharacter/AI entity
.vehicleUBlamVehicleTagDataAssetVehicle entity
.weaponUBlamWeaponTagDataAssetWeapon entity

2.2 The Model Tag Cluster

UBlamModelTagDataAsset is the root of any CVW asset's tag hierarchy:

Each sub-tag must be manually created, named to match the BLAM path exactly, and wired back to the parent model tag.

2.3 Tag-to-Asset Manager (Runtime Resolution)

FBlamTagToAssetManager maps BLAM tag indices to UE DataAssets at runtime. It enforces a strict 1:1 naming convention:

BLAM tag: objects/vehicles/banshee/banshee
UE path:  /Game/Tags/objects/vehicles/banshee/banshee-vehicle.banshee-vehicle

This coupling means renaming, moving, or restructuring assets is extremely difficult — both UE and BLAM paths must remain synchronized at all times.

2.4 Sidecar XML System (Export Path)

When exporting UE content to BLAM, the system generates a .sidecar.xml file that drives the Foundation/BLAM toolchain:

  • Contains metadata (asset name, type, output folder)
  • Defines intermediate file locations (FBX exports for models, collision, physics, animations)
  • Follows the Foundation XML schema (same format used by Halo's legacy DAM tools)
  • Generated by FBlamTagSidecarGenerator within the UE editor

3. The Core Problem

The Manual Duplication Bottleneck

The fundamental problem is that artists must manually duplicate structural data that already exists in their DCC tool (Maya) into BLAM tag fields — one field at a time, through the UE tag editor, with no automation possible.

"We already have to localize collision and render meshes under joints in maya... Tooling to help support 'scraping' the data from maya to auto-fill these out would save both time and human error."

— Senior TA, ST-9548

All of this data already exists in the Maya scene hierarchy — joints define bones, naming defines regions/permutations, and meshes are already parented. But there is no bridge to propagate it into BLAM tags.

Why Automation Doesn't Exist Today

"Due to not being able to access the actual tag fields via python, you can't hook them together in any automatic way."

— Senior TA, ST-9548

The Python scripting environment in UE cannot read or write individual tag field values. It can call high-level operations (create tag, trigger export), but cannot programmatically set the region, bone, or permutation fields that constitute the bulk of manual work.

BlamDerivedTagToAssetMapper — a class that previously provided some auto-mapping capability — was deleted in changelist 340820. Its removal left manual wiring as the only path.

The Innovation Tax

"This is a laborious process that limits innovation because the already steep cost here was one of many reasons that we would often say any new or innovative feature to be added to an asset would need to be cut."

— Ben Frazier (Creative Lead), ST-9446

The high setup cost doesn't just slow production — it actively prevents creative teams from adding features to assets. New destruction states, interaction modes, or variant systems all multiply the number of tags that must be hand-wired.

6.2 The 80/20 Finding

4 of 9 pain points trace to a single root cause: lack of Python read/write access to tag field data.

Fixing this one capability gap addresses P0 (setup time), P2 (no automation), P8 (duplicate work), and partially P6 (rename tooling). It is the single highest-leverage investment identified in this study.

7. Recommendations

Ten primary recommendations plus three additional items identified through cross-validation, organized into three tiers aligned with the TA Strategy Roadmap themes:

  • Theme 1 — Scale Through Productivity: Increase throughput without growing team
  • Theme 2 — Minimize BLAM Surface Area: Reduce UE↔BLAM coupling where possible
  • Theme 3 — Build the Foundation: Infrastructure for M2, M3, and beyond

R1: Python Tag Field Read/Write API

Short-Term PRIMARY

This is the single most important recommendation. A working prototype already exists. Completing and shipping this API unlocks the majority of automation potential across the pipeline.
AttributeDetail
SolvesP0, P2, P8 (and enables R2, R5, R8)
Effort2–3 person-months
ImpactCritical — prerequisite for all automation
RiskMedium — prototype exists but must generalize across 158 tag types with arrays/refs/enums
AI-Agent OpportunityHIGH — structured data transformations are ideal for coding agents

What it enables:

  • Automated UE Derivation setup (minutes instead of weeks)
  • Maya scene data scraping → auto-populate tag fields
  • Batch validation (catch wiring errors before export)
  • Tag diff tooling (meaningful change tracking between versions)
  • Scripted rename/refactoring operations

R2: Maya→Tag Auto-Fill Script

Short-Term Depends on R1

AttributeDetail
SolvesP8 — duplicate work between Maya and BLAM tags
Effort1–2 person-months (once R1 ships)
ImpactHigh — automates the most tedious part of Step 4 (manual wiring)
RiskLow — Maya Python API well-understood

Reads Maya scene structure (joints, meshes, naming conventions) and auto-populates collision model entries: Region, Material, Permutation, Bone, and Mesh references. Validates naming conventions before proceeding.

R3: Fix Collision Export Crash

Short-Term Quick Win

AttributeDetail
SolvesP3 — 100% repro crash when asset is loaded in level during export
Effort<1 person-month
ImpactHigh — eliminates constant workflow interruption
RiskLow — discrete, reproducible bug

Either fix the underlying concurrent access bug, or auto-unload the asset before export and reload after (transparent workaround).

R4: Auto-Export Before PIE

Medium-Term

AttributeDetail
SolvesP5 — must manually re-export to BLAM before every PIE session
Effort1–2 person-months
ImpactMedium — eliminates constant "did I remember to export?" confusion
RiskMedium — must handle export failures gracefully

Detect dirty tags (UE source data newer than BLAM binary), auto-trigger export on PIE launch, show progress. Skip if tags are up-to-date.

R5: UE Derivation Setup Wizard

Medium-Term Depends on R1 + R2

Target outcome: Reduce initial CVW asset authoring setup from 1–3 weeks to less than one day for standard asset archetypes. (Note: downstream validation/testing remains a separate effort.)
AttributeDetail
SolvesP0 — the 1–3 week initial setup bottleneck
Effort2–4 person-months
ImpactCritical — directly addresses the scaling bottleneck
RiskMedium — must handle diverse asset archetypes

A guided step-by-step workflow in the UE editor that:

  • Auto-creates all required sub-tags with correct naming
  • Auto-wires references between model → sub-tags
  • Integrates Maya auto-fill (R2) as a step
  • Pre-validates naming, folder structure, and required fields
  • Generates sidecar XML automatically

R6: Incremental Structure Generation

Strategic Requires BLAM Team Collaboration

AttributeDetail
SolvesP1 — 3–5 minutes per collision export
EffortHigh (BLAM team engagement required)
ImpactHigh — iteration speed × number of iterations = massive savings
RiskHigh — BLAM structure generator is not TA-owned

Only regenerate structures for changed regions/permutations. Cache previous results. Detect "no change" to skip entirely.

R7: Collision Preview in UE (No Export Required)

Medium-Term

AttributeDetail
SolvesP9 (partially) — slow collision feedback loop
EffortMedium (UE-side only, no engine changes needed)
ImpactHigh — instant feedback without 3–5 min export cycle
RiskMedium — rendering fidelity vs. actual BLAM collision

Render collision mesh outlines in UE viewport without requiring BLAM export. Show triangle count vs. budget in real-time. Already identified in the TA Strategy Roadmap as a Phase 1 quick win.

R8: Tag Rename/Refactoring Tool

Medium-Term Depends on R1

AttributeDetail
SolvesP6 — naming constraints and blocked renames
Effort1–2 person-months
ImpactMedium — eliminates a rare-but-painful blocker
RiskLow — pattern is well-established

Safely rename a tag across all references (UE paths, BLAM paths, scene references). Update both sides atomically. Validate no orphaned references.

R9: Collision Pipeline Redesign

Strategic

AttributeDetail

EffortVery High (multi-team, multi-quarter)

Investigation areas: Can collision be authored in UE and only exported at cook time? Can the separate mesh requirement be eliminated? Can BLAM structure generation be replaced with a UE-native equivalent? Start as investigation in M2 pre-production.

R10: Reduce Mesh Separation Overhead

Strategic

AttributeDetail

EffortHigh (touches BLAM architecture assumptions)

Medium-Term

SourceJira ST-9443; Confluence: "Stage One Asset Updating"

Warn when import order may cause marker loss. Validate marker integrity post-import. Provide one-click recovery path when markers are dropped.

R13: Tag Diff/Search/Diagnostic Tooling

Short-Term Depends on R1

AttributeDetail
SolvesNo meaningful diff between tag versions; no search within tag field blocks (ST-9548)
Effort1–2 person-months (once R1 ships)
ImpactHigh — enables change tracking, debugging, and review workflows
RiskLow — read-only tooling built on R1 API

Field-level diff between tag versions. Search across tag fields by value. Diagnostic mode showing wiring completeness and missing references.

10. Risks & Mitigations

RiskLikelihoodImpactMitigation
Python API stability across UE upgrades (5.5 → 5.7) Medium Medium Design API as thin wrapper over tag binary access. Version the API. Test on UE 5.7 eval branch early.
R5 wizard cannot handle all asset archetypes Medium Medium Start with the 3 most common archetypes (biped, vehicle, weapon). Add archetypes incrementally. Allow manual override for edge cases.
Collision pipeline redesign (R9) requires engine-level changes High High Start with UE-side preview only (no engine changes). Engage engine team for deeper changes only after quick wins prove value. Budget 2–3 months for investigation before committing.
BLAM structure generator performance (R6) is not TA-fixable High Medium Focus on "skip when unchanged" (zero-cost check). Request incremental mode from BLAM team with clear ROI data.
M2 pre-production starts before R1 ships Medium High R1 prototype already exists — accelerate. R3 (crash fix) and R7 (preview) are independent and deliver value immediately.
Team knowledge loss during M1→M2 transition High Medium Document UE Derivation workflow NOW while M1 practitioners are available. R5 wizard encodes institutional knowledge in code rather than people's heads.

What We Are NOT Recommending

  • Replacing the stitched engine — Leadership has committed to BLAM for M2/M3. We optimize within this constraint.
  • Rewriting the Tag system from scratch — Too risky, too slow. Layer automation on top.
  • Blocking M2 production for pipeline perfection — Short-term wins (R1, R3, R7) deliver value in weeks. Production can start in parallel.
  • Over-investing in collision redesign without proof — R9 starts as investigation. Commit only when quick wins (R7 preview) prove the approach.
